Learning by building with Codex

Codexを動かして 学んだこと

ただ答えを受け取るだけではなく、文脈を渡し、差分を読み、検証まで一緒に回す。 Codexを使うほど、開発は「頼み方」と「確かめ方」の練習でもあると分かってきました。

学びを見る 進め方を見る

Codexは、速く書くためだけの道具ではありませんでした。自分の考えを言語化し、 変更の根拠を読み、動作確認まで含めて小さく進めるための相棒です。

Key learnings

使って分かった5つのこと

便利さの中心にあるのは、自動化そのものよりも、曖昧な作業を小さな判断に分けられることでした。

01

文脈を先に渡す

目的、前提、触ってよい範囲、期待する確認方法を先に置くと、返ってくる提案の粒度が揃います。

02

小さく任せる

大きな変更を一気に頼むより、調査、実装、確認を分けた方が読みやすく、戻しやすくなります。

03

差分で理解する

生成されたコードを完成品として眺めるより、なぜその差分になったかを追う方が身につきます。

04

検証まで含める

テスト、ブラウザ確認、ログ確認まで頼むと、会話が感想ではなく事実ベースに変わります。

05

失敗を素材にする

うまくいかなかった出力も、条件の抜け、前提の誤り、確認不足を見つける材料になります。

Working loop

いちばん効いた進め方

Codexとの作業は、依頼して終わりではなく、短い往復を重ねるほど精度が上がりました。

  1. 1

    現状を見せる

    ファイル構成、エラー、直したい振る舞いを共有して、まず観察してもらう。

  2. 2

    目的を絞る

    「何を良くしたいか」を一つにして、変更範囲が広がりすぎないようにする。

  3. 3

    差分を読む

    実装の意図、既存コードとのつながり、残ったリスクを自分の言葉で確認する。

  4. 4

    動かして確かめる

    テストや画面確認を通して、納得ではなく事実で終わらせる。

Notes

場面ごとの気づき

同じCodexでも、依頼、修正、検証で効く言葉が変わります。下の切り替えで、使い分けを整理しました。

よい依頼は、小さな設計書になる

「いい感じにして」よりも、目的、読者、制約、確認方法を渡す方が、Codexは迷わず動けます。 依頼文を書く時間は、そのまま自分の設計を整える時間になりました。 

目的: 初見でも分かる静的サイトにする
前提: 既存ファイルはまだない
変更範囲: HTML / CSS / JS / 画像アセット
確認: ブラウザで崩れと読みやすさを見る

修正依頼は、違和感を具体化する練習

「変」だけでは直せません。どの画面幅で、どの要素が、どんな期待とズレているかを言うと、 修正は速くなります。これは人間同士のレビューにもそのまま効きました。 

現象: モバイルで見出しが窮屈
期待: 2行までは自然に読める
範囲: ヒーローだけ調整
避けたいこと: 全体の余白を崩さない

検証は、AIと自分を同じ現実に戻す

画面、テスト、ログ、差分を見ると、会話の雰囲気ではなく動いている事実で判断できます。 Codexはここまで一緒に回して初めて、安心して任せられる道具になります。 

確認したこと:
- 主要セクションが表示される
- タブが切り替わる
- モバイル幅で文字が重ならない
- 画像アセットが読み込まれる

一番の収穫

Codexを使うほど、コードを書く力だけでなく、問題を分ける力、説明する力、確かめる力が鍛えられる。
